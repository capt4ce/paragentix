Evidence from the codebase

Backend concentration

internal/board/app.go: 1,184 LOC. It mixes schema, HTTP, auth, jobs, events, settings, scheduler and process execution.

Frontend concentration

frontend/src.tsx: 754 LOC. Unmounted LegacyApp duplicates active UI; active App owns routing, data, dialogs and rendering.

UI foundations absent

No shadcn, Radix, Tailwind, aliases or semantic tokens. Global CSS uses repeated literal colors.

Ordered implementation sequence

  1. Baseline: preserve passing Go tests/vet; add failure-path tests before behavior changes; pin frontend versions.
  2. Delete: remove proven dead old/legacy handlers and unmounted LegacyApp.
  3. Move only: split board package by auth, jobs, events, scheduler, migrations and HTTP without changing APIs.
  4. Correctness: harden transaction/query/process errors, then extract only proven repeated scanners/lookups.
  5. Frontend move only: create focused App, route/API, board, job, notification and workspace modules with typed boundaries.
  6. shadcn foundation: semantic tokens, aliases and utilities; migrate primitive families incrementally.
  7. Design rollout: apply only the approved visual direction, beginning with the blocked-job intervention workflow.

shadcn/ui migration boundary

Use shadcn for accessible Button, Input, Textarea, Label, Card, Badge, Dialog, Dropdown Menu, Tabs, Select and Checkbox. Keep board lanes, horizontal scrolling, timeline, responsive sheets and agent-specific compositions as product code. Preserve query-string routing and the Vite base path during migration.

Validation

  • go test ./... and go vet ./... after every backend slice.
  • Frontend tests, typecheck, production build and embedded-asset smoke test.
  • Keyboard/focus validation for dialogs, menus and tabs; state must not depend on color.
  • Viewport checks at 320, 375, 414, 768, 1024 and 1440 px.
  • Route restoration, invitations, pagination, SSE cleanup, board selection and mobile lane navigation.

Approval decisions

  1. Choose Mission Control, Workshop Ledger or Agent Graph.
  2. Decide whether light mode ships in the first implementation.
  3. If Agent Graph wins, decide initial scope versus later optional view.

Recommendation: Mission Control best matches the current product’s agent-supervision core while keeping implementation risk below Agent Graph.
