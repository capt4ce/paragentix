Workspaces become collaborative. Projects own directories.

A minimal extension of the existing Go + SQLite + React app: workspace memberships and invitations, explicit Projects, and a required Project on every Column.

Problems captured

Requested outcomes

  1. Avatar menu closes on outside click and option selection.
  2. Workspace creation has only a Name.
  3. Projects belong to Workspaces and own their directory.
  4. Workspaces list and tabbed detail UI.
  5. Member list, removal confirmation, and email invitations.
  6. Each Column selects one Project; one Project may serve many Columns.

Product model

EntityResponsibilityRelationship
WorkspaceName + collaborative accessMany members, Projects, and Boards
ProjectName + project directoryOne Workspace; no direct Project association
ColumnExecution laneOne Board + one Project in the same Workspace
InvitationHashed, expiring email inviteOne Workspace and recipient email

Existing owner rows are backfilled into memberships. Existing workspace directories become a deterministic Default Project during migration.

Feature slices

1. Avatar behavior

Keep native <details>; close via one outside-pointer listener and before every menu action.

2. Workspace directory

Accessible-workspace list plus Info, Projects, and Users tabs. Owners edit via existing modal patterns.

3. Boards tab

List every Board directly belonging to the Workspace with Column count and Open action. Boards remain Workspace-only and receive no Project foreign key.

4. Collaboration

Membership-scoped APIs; owners invite any email and remove members, with final-owner protection.

5. Project Columns

Required native Project select in Column create/update; cross-workspace project IDs rejected server-side. Existing Git worktree and Worktree name fields remain unchanged.

Implementation order

  1. Migration tests and restart-safe schema additions.
  2. Shared membership/owner authorization helpers.
  3. Workspace, Project, Board-list, member, and invitation APIs.
  4. Stdlib SMTP behind a tiny fakeable mail seam, only if no mail transport already exists.
  5. Column project_id validation and DTO updates; no Board-to-Project association.
  6. Workspaces UI, Info/Projects/Boards/Users tabs, shared modals, and avatar dismissal.
  7. Go tests/build, frontend test/build, authenticated API and browser smoke.
  8. Deploy embedded frontend and service, verify public, then commit and push.

Safeguards

  • Membership is the API boundary; UI hiding is not authorization.
  • Invitation tokens are random and stored hashed; secrets never logged.
  • Project directory canonicalization remains inside global WORKSPACE_ROOT.
  • Final owner cannot be removed.
  • Existing uncommitted frontend changes are preserved and reconciled.
