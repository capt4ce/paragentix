Reliable artifact handoff between sequential jobs

An explicit, durable manifest connects one job’s files to the next job’s prompt—without copying blobs, parsing prose, or adding a general dependency graph.

Current evidence

  • The scheduler already serializes jobs per lane and selects the first todo job by position.
  • The initial Hermes prompt contains project context, task, and done definition—but no predecessor output.
  • Final responses are stored as event text and run summaries, not structured handoffs.
  • The job schema and UI have no artifact metadata.

Proposed approach

Job A
Writes files into its selected project/worktree and publishes paths + note.
→
Scheduler
Loads nearest completed predecessor and validates paths against the effective root.
→
Job B
Receives an “Upstream handoff” prompt section and reads only what it needs.
Smallest rung that holds

Keep column order as the dependency model. Store metadata in SQLite; keep content in the repository/worktree. Do not add blob storage, response parsing, or a cross-column DAG in V1.

Scoped assumptions

  1. Consume only the nearest earlier completed job in the same column.
  2. Artifacts are existing project-relative files or directories.
  3. Missing or unsafe paths block the consumer instead of silently losing context.
  4. Jobs with no manifest remain unchanged.

Implementation tasks

1. Persist the manifest

internal/board/migrations.go, internal/board/app.go, internal/board/app_test.go

Add idempotent columns, model fields, JSON scanning, and migration coverage.

2. Validate through the existing job API

internal/board/jobs.go, internal/board/app_test.go

Extend PATCH/read responses. Reject absolute paths, traversal, empties, duplicates beyond normalization, excessive entries, and oversized notes. Lock completed manifests.

3. Inject the predecessor handoff

internal/board/scheduler.go, internal/board/app_test.go

Select the nearest lower-position completed job, resolve paths under the effective project/worktree root, reject symlink escapes, block on missing artifacts, and append one deterministic prompt section.

4. Expose the contract in the inspector

frontend/src/lib/types.ts, frontend/src/App.tsx, frontend/src.test.ts

Add a “Handoff to next job” section with newline-separated paths, a concise note, guidance, save/error states, and read-only completed-job display. Preserve existing task, comments, retry, input, and archive flows.

5. Verify end to end

README.md plus existing test suites

Run Go and frontend suites, then a disposable two-job smoke test. Prove successful read, missing-file block, symlink-escape block, retry determinism, and legacy no-manifest behavior. Commit and push after approved implementation.

Acceptance criteria

  • A completed job exposes up to 20 relative paths and a concise note.
  • The immediately following job receives the handoff automatically.
  • SQLite stores metadata only; artifact contents remain in the selected root.
  • Absolute paths, traversal, symlink escapes, and missing files cannot start a consumer with false context.
  • Retries do not duplicate handoff sections.
  • Legacy jobs still run.
  • Go tests, frontend tests/build, and a real two-job smoke test pass.

Risks and deliberate cuts

  • Different worktrees: V1 blocks if the consumer cannot see the producer path; it does not copy across roots.
  • Untrusted prose: final output is not parsed into a manifest.
  • Fan-in/fan-out: deferred until a real cross-column use case justifies a dependency table.
  • Large files: paths are passed, not contents; the consumer reads selectively.

Open questions for approval

  1. Keep completed manifests immutable, or allow repair after completion?
  2. Is immediate predecessor enough for V1, or are cross-column upstream selections required now?
  3. Accept directories as artifacts, or files only?
