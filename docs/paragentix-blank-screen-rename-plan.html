Problems captured

  1. A newly created account reaches a blank screen.
  2. The entire project identity must become Paragentix: directory, code identity, binary, service, nginx route, and public path.

Root cause confirmed

Fresh signup and session checks succeed. The first lane response contains:

[{"name":"Lane 1","jobs":null}]

The React board calls l.jobs.map(...). Go serializes an uninitialized slice as null, causing a render-time exception only for fresh, empty accounts. Existing accounts with jobs bypass this state. Assets and routes return HTTP 200, so this is not an nginx or Vite asset failure.

Root fix: honor the API collection contract and emit "jobs": [].

Implementation sequence

1 · Regression first

Add a backend test proving a fresh account’s empty lane returns an array, then confirm it fails.

2 · Minimal contract fix

Initialize Lane.Jobs as []Job{} before scanning. Run focused and full Go tests.

3 · Source identity rename

Rename labels, module/imports, command directory, binary, Vite base, docs/config references, and tracked paths from custom-agent to paragentix.

4 · Build before moving runtime

Run frontend tests/build and Go tests/build. Confirm generated assets use /projects/paragentix/.

5 · Runtime migration

Stop the old service; move the project directory with SQLite DB/WAL/SHM together; install paragentix.service; update nginx; retain an old-path redirect; validate and restart.

6 · Public proof and delivery

Create a fresh account through the public route, verify the empty board renders, check every asset/API response, then commit and push if a git remote exists.

Acceptance criteria

  • Fresh account sees an empty board—not a blank screen.
  • Empty lane JSON contains "jobs": [].
  • UI and tracked source display Paragentix.
  • Directory, module/imports, binary, service, nginx route, and public URL use paragentix.
  • Go tests, frontend tests, and production builds pass.
  • /projects/paragentix/ and assets return 200.
  • Old path redirects to the new one.
  • Commit and push complete when a remote is configured.

Safeguards

  • Stop the process before moving SQLite and its WAL/SHM files.
  • Build before runtime path changes to minimize downtime.
  • Run nginx -t before reload.
  • Keep the old unit/route until new service and public smoke checks pass.
  • Never claim a push if no remote is configured.

Implementation gate

This is a review plan only. No application, runtime, deployment, or identity changes have been made. Implementation starts only after explicit approval.
